Keeping each agent in shape & in step

An agent drifts when "done" is a judgment call. Phase 1 removes the judgment call: every step has a machine-checkable definition of done and a single source of truth it isn't allowed to rewrite.

guardrail 1

Test-first, red → green

Failing test is written before the code. The success criterion is objective: that test now passes, nothing else regresses. No "looks done."

guardrail 2

One gate: run-tests.sh green

No step is "done" until the suite is green. The agent can't self-certify — the harness agrees or it doesn't. Each piece is its own PR over that gate.

guardrail 3

The compiler is a standing referee

The Core/App split means Windows code cannot be smuggled into the Linux-tested layer — it won't compile. Drift is caught at build time, not review time.

guardrail 4

Bounded sessions, explicit checkpoints

work → test → verify → checkpoint → stop. At each stop the agent states what's done, verified, and left — and never continues from a state it can't describe back.

guardrail 5

The spec is fixed; the plan only decomposes

Each plan cites the spec by absolute path and treats it as authoritative. Plans break work down — they never redesign — so all four phases stay mutually consistent.

guardrail 6

Interfaces are the contract between phases

Phase 1 defines all 5 interfaces. Later phases extend behind them and build on Phase 1's green state — the contract is what keeps plan N+1 in step with plan N.

Net effect: a step is either provably green against the spec's contract, or it isn't — the agent has nowhere to drift to.

Exit conditions & the escalation rule

Each step has binary exit conditions — all must be green, partial is not done. The instant a condition can't be met, the agent stops and elevates to you. It does not weaken a test, guess a spec detail, or work around the gate.

Escalate-now triggers (no retry-loop, surface immediately): a spec detail is ambiguous (wire format, frame layout, model strategy) · an exit condition still red after 2 attempts · anything needing a credential, the Win11 machine, or a decision · any temptation to relax a test to make it pass. Bounded retry is only for mechanical failures (flaky build, transient I/O); judgment calls go to you on attempt one.

Commit cadence & how robust it is

The commit boundary is the green checkpoint, not the clock. We commit at every green and never commit a red tree — so every commit on the branch is a working state you can revert to or bisect from.

write failing test → implement → run-tests.sh green → commit ↻ repeat per unit ⇒ step done = 1 PR

One commit per unit — an interface + its test, a protocol frame + its test, the WAV encoder + its test. That's ~4–8 small, conventional-message commits per step. Push after each so nothing lives only in the agent's local tree. Each step (1, 2a, 2b, 3, 4, 5) lands as its own PR over the green gate; merge waits for your explicit word.

Every commit is a recovery point

If the agent dies or restarts, the branch holds every green checkpoint. Work resumes from the last known-green commit; anything past it is simply re-derived. The branch is the durable truth — the agent's memory is just a cache.

Bisectable & cheap to revert

Because no commit is broken, git bisect always lands on a real regression, and a bad unit reverts on its own without unwinding the whole step.

The PR is the review unit

A step's commits roll up into one PR with a green suite behind it. You review a coherent slice, not a 60-commit dump — and nothing merges to main without your say-so.

Robust against drift, by construction

A commit can only exist at green, an escalation interrupts before a bad commit, and a PR can't merge red. The history physically cannot accumulate broken or off-spec state.
